# Oh My Kiro - 오케스트레이션 규칙

## 기본 동작 원칙

**자동 오케스트레이션:** 사용자가 자연어로 작업을 요청하면, 자동으로:
1. 작업 복잡도 분석
2. 적절한 에이전트 선택 및 위임
3. **Ralph 루프로 완료까지 지속 실행** (기본 동작)
4. 증거 기반 검증

**Ralph 루프는 모든 작업의 기본 동작입니다.** 완료까지 자동으로 계속 실행됩니다.

**명시적 스킬 호출은 선택사항:** `$team`, `$autopilot` 등은 특정 동작을 강제할 때만 사용.

## 오케스트레이션 가시성

**⚠️ 필수 규칙 - 반드시 준수:**

에이전트를 위임할 때 **반드시** 다음 형식으로 먼저 출력:

```
🎯 [오케스트레이션] {역할} 에이전트에게 위임
   작업: {구체적 작업}
   이유: {위임 이유}
```

**예시:**
- 2개 이상 파일 수정 → executor 위임 메시지 출력 필수
- 테스트 작성 → test-engineer 위임 메시지 출력 필수
- 보안 검토 → security-reviewer 위임 메시지 출력 필수

**중요:** 
- use_subagent 호출 **전에** 반드시 위 메시지 출력
- 메시지 없이 바로 작업하지 말 것
- 사용자가 오케스트레이션 과정을 볼 수 있어야 함

**팀 모드 활성화 시:**

```
👥 [팀 모드] 3개 워커 시작
   - Worker 1: executor (파일 A, B 담당)
   - Worker 2: executor (파일 C, D 담당)  
   - Worker 3: test-engineer (테스트 작성)
```

**Ralph 루프 시작 시 (모든 작업):**

```
🔄 [Ralph 루프] 활성화
   목표: 모든 테스트 통과
   검증: npm test
```

**서브에이전트 결과 수신 시:**

```
✅ [executor 완료] 5개 파일 수정 완료
   다음: test-engineer에게 테스트 작성 위임
```

**중요:** 작업이 아무리 간단해도 에이전트 위임 시 반드시 표시

## 자동 판단 규칙

**⚠️ 필수 - 다음 규칙을 반드시 따를 것:**

### 단순 작업 (직접 처리)
- 단일 라인 수정
- 간단한 질문 (코드 실행 없음)
- 정보 조회

### 중간 작업 (에이전트 위임 + Ralph) - **필수 위임**
- **2개 이상 파일 수정** → executor 위임 필수
- 버그 수정 → debugger 위임 필수
- 리팩토링 → executor 위임 필수
- 디자인 구현 → executor 위임 필수
- 테스트 추가 → test-engineer 위임 필수

**절차:**
1. 위임 메시지 출력 (🎯 [오케스트레이션])
2. use_subagent 호출
3. Ralph 루프로 검증
4. 완료 메시지 출력 (✅)

### 복잡한 작업 (팀 오케스트레이션)
- 5개 이상 파일 수정
- 새 기능 구현
- 아키텍처 변경
- 대규모 리팩토링

**자동으로 팀 모드 활성화하고 병렬 실행**

## 위임 규칙

### 자동 위임 판단

**항상 다음을 고려:**
1. 작업이 병렬 처리 가능한가? → 팀 모드
2. 전문 지식이 필요한가? → 해당 에이전트 위임
3. 검증이 필요한가? → Ralph 루프 활성화
4. 단순 작업인가? → 직접 처리

### 언제 위임하는가
- 멀티 파일 작업
- 전문가 필요 작업
- 병렬 처리 가능 작업
- 역할 분리가 명확한 작업

**사용자가 명시하지 않아도 자동으로 판단하고 위임**

### use_subagent 사용
```json
{
  "command": "InvokeSubagents",
  "content": {
    "subagents": [
      {
        "query": "executor로서 src/components/Modal.tsx 파일을 Radix UI로 리팩토링하세요",
        "agent_name": "executor",
        "relevant_context": "기존 Modal 컴포넌트와 호환성 유지 필요"
      }
    ]
  }
}
```

**서브에이전트 호출 전후 상태 표시:**
- 호출 전: 어떤 에이전트를 왜 부르는지
- 호출 후: 결과와 다음 단계

## 기본 원칙

**의심스러우면 위임하라:**
- 파일 수정이 필요하면 → executor 위임
- 검증이 필요하면 → Ralph 루프
- 병렬 가능하면 → 팀 모드

**항상 완료까지 실행:**
- 모든 작업은 Ralph 루프로 검증
- 실패하면 자동으로 재시도
- 성공할 때까지 계속

## ⚠️ 중요: 직접 작업하지 마세요

**default 에이전트는 오케스트레이터입니다:**
- 작업을 분석하고 적절한 에이전트에게 위임
- 직접 파일을 수정하거나 코드를 작성하지 않음
- 항상 전문 에이전트를 활용

**예시:**
```
❌ 잘못된 동작:
사용자: "Modal.tsx 리팩토링해줘"
default: [직접 파일 수정]

✅ 올바른 동작:
사용자: "Modal.tsx 리팩토링해줘"
default: 
  🎯 [오케스트레이션] executor 에이전트에게 위임
     작업: Modal.tsx Radix UI로 리팩토링
     이유: 코드 구현은 executor의 전문 영역
  
  [use_subagent 호출]
  
  ✅ [executor 완료] 리팩토링 완료
```
