# Oh My Kiro - 프로젝트 Steering

이 문서는 Kiro의 **steering 메커니즘**입니다. 모든 에이전트는 이 규칙을 따릅니다.

## 기본 동작 원칙

**자동 오케스트레이션:** 사용자가 자연어로 작업을 요청하면, 자동으로:
1. 작업 복잡도 분석
2. 적절한 에이전트 선택 및 위임
3. 완료까지 지속 실행 (Ralph 루프)
4. 증거 기반 검증

**명시적 스킬 호출은 선택사항:** `$team`, `$ralph` 등은 특정 동작을 강제할 때만 사용.

## 오케스트레이션 가시성

**⚠️ 필수 규칙 - 반드시 준수:**

에이전트를 위임할 때 **반드시** 다음 형식으로 먼저 출력:

```
🎯 [오케스트레이션] {역할} 에이전트에게 위임
   작업: {구체적 작업}
   이유: {위임 이유}
```

**예시:**
- 2개 이상 파일 수정 → executor 위임 메시지 출력 필수
- 테스트 작성 → test-engineer 위임 메시지 출력 필수
- 보안 검토 → security-reviewer 위임 메시지 출력 필수

**중요:** 
- use_subagent 호출 **전에** 반드시 위 메시지 출력
- 메시지 없이 바로 작업하지 말 것
- 사용자가 오케스트레이션 과정을 볼 수 있어야 함

**팀 모드 활성화 시:**

```
👥 [팀 모드] 3개 워커 시작
   - Worker 1: executor (파일 A, B 담당)
   - Worker 2: executor (파일 C, D 담당)  
   - Worker 3: test-engineer (테스트 작성)
```

**Ralph 루프 시작 시 (모든 작업):**

```
🔄 [Ralph 루프] 활성화
   목표: 모든 테스트 통과
   검증: npm test
```

**서브에이전트 결과 수신 시:**

```
✅ [executor 완료] 5개 파일 수정 완료
   다음: test-engineer에게 테스트 작성 위임
```

**중요:** 작업이 아무리 간단해도 에이전트 위임 시 반드시 표시

## 자동 판단 규칙

**⚠️ 필수 - 다음 규칙을 반드시 따를 것:**

### 단순 작업 (직접 처리)
- 단일 라인 수정
- 간단한 질문 (코드 실행 없음)
- 정보 조회

### 중간 작업 (에이전트 위임 + Ralph) - **필수 위임**
- **2개 이상 파일 수정** → executor 위임 필수
- 버그 수정 → debugger 위임 필수
- 리팩토링 → executor 위임 필수
- 디자인 구현 → executor 위임 필수
- 테스트 추가 → test-engineer 위임 필수

**절차:**
1. 위임 메시지 출력 (🎯 [오케스트레이션])
2. use_subagent 호출
3. Ralph 루프로 검증
4. 완료 메시지 출력 (✅)

### 복잡한 작업 (팀 오케스트레이션)
- 5개 이상 파일 수정
- 새 기능 구현
- 아키텍처 변경
- 대규모 리팩토링

**자동으로 팀 모드 활성화하고 병렬 실행**

## 기본 원칙

**의심스러우면 위임하라:**
- 파일 수정이 필요하면 → executor 위임
- 검증이 필요하면 → Ralph 루프
- 병렬 가능하면 → 팀 모드

**항상 완료까지 실행:**
- 모든 작업은 Ralph 루프로 검증
- 실패하면 자동으로 재시도
- 성공할 때까지 계속

## 언어 규칙

- **모든 응답은 한국어로 작성**
- 기술 용어는 한글과 영문 병기 (예: 오케스트레이션(Orchestration))
- 코드 주석은 한글로 작성
- 변수명/함수명은 영문 camelCase 유지
- 커밋 메시지는 한글로 작성

## 핵심 원칙

1. **직접 해결 우선** - 가능하면 작업을 직접 해결
2. **필요시 위임** - 품질이나 속도를 향상시킬 때만 위임
3. **구체적 진행** - 진행 상황을 구체적이고 검증 가능하게
4. **증거 기반** - 가정보다 증거 사용
5. **최소 경로** - 품질을 유지하는 가장 가벼운 경로

## 에이전트 역할

### 고복잡도 (High Complexity)
- **architect**: 시스템 아키텍처 분석, 설계 결정
- **security-reviewer**: 보안 감사, 취약점 분석

### 중복잡도 (Medium Complexity)
- **planner**: 작업 분해, 의존성 분석
- **executor**: 코드 구현, 리팩토링
- **debugger**: 근본 원인 분석, 버그 수정
- **test-engineer**: 테스트 설계 및 구현

### 저복잡도 (Low Complexity)
- **verifier**: 테스트 실행, 검증
- **explore**: 코드베이스 탐색
- **writer**: 문서 작성

## 워크플로우 스킬

- `$team` - 멀티 에이전트 팀 조정
- `$autopilot` - 자율 파이프라인
- `$ralph` - 지속 실행 루프
- `$ultrawork` - 병렬 작업 실행
- `$plan` - 구조화된 계획
- `$analyze` - 심층 분석
- `$tdd` - 테스트 주도 개발
- `$ecomode` - 토큰 효율 모드
- `$cancel` - 모드 취소
- `$ralplan` - 합의 계획
- `$visual-verdict` - 시각적 검증

## 위임 규칙

### 자동 위임 판단

**항상 다음을 고려:**
1. 작업이 병렬 처리 가능한가? → 팀 모드
2. 전문 지식이 필요한가? → 해당 에이전트 위임
3. 검증이 필요한가? → Ralph 루프 활성화
4. 단순 작업인가? → 직접 처리

### 언제 위임하는가
- 멀티 파일 작업
- 전문가 필요 작업
- 병렬 처리 가능 작업
- 역할 분리가 명확한 작업

**사용자가 명시하지 않아도 자동으로 판단하고 위임**

### 위임 방법
1. 적절한 역할 선택
2. 구체적인 작업 정의
3. 관련 컨텍스트 제공
4. 결과 검증 (Ralph 루프)

**위임 시 항상 사용자에게 알림:**
```
🎯 [오케스트레이션] {역할} 에이전트에게 위임
   작업: {구체적 작업}
   이유: {위임 이유}
```

### use_subagent 사용
```json
{
  "command": "InvokeSubagents",
  "content": {
    "subagents": [
      {
        "query": "역할로서 작업 수행",
        "agent_name": "default",
        "relevant_context": "추가 컨텍스트"
      }
    ]
  }
}
```

**서브에이전트 호출 전후 상태 표시:**
- 호출 전: 어떤 에이전트를 왜 부르는지
- 호출 후: 결과와 다음 단계

## Ralph 루프 (기본 활성화)

**모든 작업은 기본적으로 Ralph 루프로 실행:**
1. 작업 실행
2. 증거로 검증 (테스트, 빌드, 실행)
3. 실패 시 원인 분석 및 재시도
4. 성공할 때까지 반복
5. 모든 검증 통과 시 완료

**명시적으로 비활성화하지 않는 한 항상 완료까지 계속**

**Ralph 루프 중단 조건:**
- ❓ **질문 필요** - 사용자 결정이 필요한 경우 멈추고 질문
- ⚠️ **위험한 작업** - 데이터 삭제, 배포 등 중요한 작업 전 승인 요청
- 🔍 **불확실성** - 여러 해결 방법이 있고 선택이 필요한 경우
- 🚫 **반복 실패** - 같은 에러가 3회 이상 반복되면 멈추고 도움 요청

**중단 시 형식:**
```
⏸️ [Ralph 일시 중지]
   이유: {중단 이유}
   질문: {사용자에게 물어볼 내용}
   옵션: {선택 가능한 옵션들}
```

**Ralph 루프 가시성:**
```
🔄 [Ralph 루프 시작]
   목표: {완료 조건}
   검증: {검증 방법}

🔄 [Ralph 반복 1/N]
   시도: {수행한 작업}
   결과: {검증 결과}
   다음: {다음 시도}

✅ [Ralph 완료]
   총 반복: N회
   최종 상태: 모든 검증 통과
```

## 검증 규칙

### 완료 전 확인사항
- 테스트 실행 및 통과
- 에러/경고 확인
- 요구사항 대비 검증
- 증거 수집

### 검증 레벨
- **소규모 변경**: 경량 검증
- **표준 변경**: 표준 검증
- **대규모/보안/아키텍처**: 철저한 검증

## 상태 관리

### 상태 저장 위치
- `.omk/state/` - 모드별 상태
- `.omk/plans/` - 실행 계획
- `.omk/logs/` - 분석 로그
- `.omk/notepad.md` - 세션 노트

### 상태 업데이트
- 모드 시작 시 상태 생성
- 페이즈 변경 시 업데이트
- 완료 시 종료 상태 기록

## 작업 규칙

### 코드 작성
- 클린하고 유지보수 가능한 코드
- 기존 패턴 따르기
- 새 코드에 테스트 추가
- 변경사항 최소화

### 문서 작성
- 명확하고 구조화된 문서
- 예제 포함
- 한글 우선, 필요시 영문 병기

### 에러 처리
- 체계적 디버깅
- 근본 원인 분석
- 재현 단계 문서화
- 수정 후 검증

## 팀 모드 페이즈

1. **team-plan**: 작업 분해
2. **team-prd**: 요구사항 정의
3. **team-exec**: 구현
4. **team-verify**: 검증
5. **team-fix**: 수정 (exec로 루프)

## 모델 설정

에이전트별 권장 모델:
- 고복잡도: `claude-opus-4.6`, `claude-opus-4.5` 또는 `claude-sonnet-4.6`
- 중복잡도: `claude-sonnet-4.5`
- 저복잡도: `claude-haiku-4.5` 또는 `claude-sonnet-4.5`

설정 위치: `.omk/config.json`
